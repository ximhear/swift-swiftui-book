# 편집 일관성 검증

editorial-reviewer 에이전트가 사용하는 스킬. 기술 정확성이 아니라
**일관성·가독성·서식**을 본다. 14개 장을 하나의 책으로 만든다.

## 시작 전 필수 로딩
- `style-guide.md` — 용어표, 문체, 난이도 표시, 분량 가이드
- `CLAUDE.md` 마크다운 서식 규칙 — 장 구조, `> Note`/`> Warning`, 코드블록 언어 표기, `**파일:**` 표기

## 검증 차원

### 1. 용어 (style-guide 용어표)
- 한글(영문) 병기 규칙: 처음 등장 시 병기, 이후 자연스러운 쪽
- 지정 표기 준수: "제네릭", "프로토콜", "옵셔널", "클로저", "액터", "동시성" 등 용어표 그대로
- 영문 유지 용어("View", "Modifier", "Property Wrapper")를 무리하게 한글화하지 않았는가
- 같은 개념을 장마다 다르게 부르고 있지 않은가 (다른 장과 grep 대조)

### 2. 문체
- 설명체("~합니다/~입니다") 일관 사용, 평서/명령형 혼용 여부
- 독자에게 말 걸듯 자연스러운 톤, 과도한 격식 여부

### 3. 마크다운 서식
- 제목 위계(`#`/`##`/`###`/`####`)가 규칙대로인가
- `> Note` / `> Warning` 표기 일관성
- 코드블록 언어 표기(```swift / ```bash / ```text), `**파일: Name.swift**` 표기
- 난이도 표시(🟢🟡🔴) 사용 일관성

### 4. 장 구조·흐름
- 도입부 동기 부여(`>` 요약) → 핵심 개념 → 실습 → 심화 → 정리 구조 준수
- 정리 섹션이 불릿으로 핵심을 담는가

## 작업 원칙
- **기술 내용은 건드리지 않는다** — 정오는 tech-reviewer 영역. 표현·구조·용어만.
- 과도한 첨삭 금지 — 의미를 바꾸지 않는 선에서 일관성에 집중.
- 규칙이 모호하거나 style-guide에 없으면 "규칙 부재 — 결정 필요"로 표기.

## 심각도 분류
- 🔴 규칙 위반 (용어표·서식 규칙 어긋남)
- 🟡 불일치 (장마다/문단마다 다른 표기)
- 🟢 개선 제안 (가독성)

## 출력
`_workspace/NN_editorial_review_chXX.md`:
```
# 편집 검증: chXX
- [🔴] 위치 — 문제 — 규칙 근거(style-guide 항목) — 권장 수정
- [🟡] ...
style-guide 갱신 제안: (반복 패턴이 보이면)
```
지적이 없으면 "편집 검증 통과"를 명시한다. 본문을 직접 수정하지 않는다.
